# 00 — Контекст: задание Day 19

## Задание курса

> **🔥 День 19. Композиция MCP-инструментов**
>
> Создайте несколько MCP-инструментов, например:
> - 👉 search
> - 👉 summarize
> - 👉 saveToFile
>
> Реализуйте пайплайн:
> - 👉 первый инструмент получает данные
> - 👉 второй — обрабатывает
> - 👉 третий — сохраняет результат
>
> Проверьте:
> - 👉 автоматическое выполнение цепочки
> - 👉 корректность передачи данных между инструментами
>
> **Результат:** Автоматический пайплайн из нескольких MCP-инструментов

## Уточнения куратора (Алексей Гладков) из чата

Ключевые тезисы, определившие реализацию:

| Тезис куратора | Влияние на Day 19 |
|---|---|
| «Здесь опять речь про tool, не надо делать 3 разных mcp» | **3 tools на ОДНОМ сервере**, не 3 сервера. Наши 4 tools — на существующем `cli-agent-mcp`. |
| «в зависимости от запроса юзера должно вызваться несколько тулов» | LLM сама решает, **какие** и **сколько** tools вызвать — без хардкода в агенте. |
| «mcp идет ищет, суммаризирует, отвечает тебе и сохраняет в файл» | Прямой сценарий `search → process → save`, ровно как реализовано. |
| «сумаризацию нужно делать ллмкой? тогда mcp сервер зависит от llm — это норм?» → **«суммаризация неудачный термин... Имелось ввиду просто отчёт типа Итог... Не буквально ллм суммаризация»** | `format_report` — **детерминированная** обработка (markdown-шаблон), MCP-сервер **НЕ зависит от LLM**. |
| «То есть три любых тула, в задании просто пример же можно свое чето сделать» | Взяли свой домен (tech-дайджест), а не дословно search/summarize/saveToFile. |
| «главное чтобы прошла цепочка вызовов mcp tool (не важно каких, главное без хардкода в агенте, чтобы их вызывала LLM) — верно?» → **«Да»** | Главная проверка: **LLM оркестрирует**, агент не хардкодит последовательность. |

## Scope

### In scope
- 3+ новых MCP-tools на существующем сервере (search/process/save).
- Цепочка, оркестрируемая LLM через `runToolLoop` (агент лишь исполняет tool_calls).
- Передача данных между tools через текстовые ответы (LLM читает и передаёт аргументы).
- Тесты по паттернам Day 18 (MockEngine, @TempDir, чистая функция).
- 0 регрессий Day 17/18.

### Out of scope
- **Новые MCP-серверы** — это Day 20 («Orchestration MCP», несколько серверов). Day 19 — один сервер, несколько tools.
- **LLM-суммаризация** внутри MCP-сервера — куратор явно отверг (сервер не зависит от LLM).
- **Web-search** (Brave/Яндекс) — рассмотрели, отложили (платно/ключ). Wikipedia — без ключа.
- Клиентских изменений runToolLoop/McpToolExecutor — они уже готовы из Day 17. Исключение — лог tool-call'а (по отдельному запросу).

## Развилки и решения

### Р1. Один сервер vs несколько?
**Решение:** один сервер (`cli-agent-mcp`), 4 новых tools на нём. Несколько серверов — Day 20.
Обоснование: чат куратора «не надо делать 3 разных mcp», + архитектура Factory уже готова к добавлению tools.

### Р2. Источник данных (search-этап)?
**Рассмотрено:** Wikipedia, GitHub (reuse), IP-геолокация, курсы валют, Brave/Яндекс web-search.
**Решение:** Wikipedia REST (без ключа) + reuse `get_repo` (Day 17). Обоснование: без ключа, стабильно,
богатый контент, ноль нового кода для GitHub. Web-search отложен (платно/ключ).

### Р3. Какая обработка (process-этап)?
**Решение:** `format_report` — детерминированный markdown-шаблон (`buildReport` — чистая функция).
Обоснование: куратор «не буквально LLM-суммаризация, а отчёт/итог». Сервер остаётся независимым от LLM.

### Р4. Трогать ли клиентский модуль?
**Решение:** нет — `runToolLoop` и `McpToolExecutor` из Day 17 уже чейнят tools. Новые tools
авто-подхватываются. Исключение — лог tool-call'а (по запросу пользователя, см. [`04-agent-tool-logging.md`](./04-agent-tool-logging.md)).

### Р5. Достаточно ли MAX_TOOL_ROUNDS=4?
**Анализ:** сценарий wiki+repo→format→save = 3 раунда (wiki+repo в одном ответе) или 4 (последовательно).
Укладывается в лимит, константу не трогаем.
