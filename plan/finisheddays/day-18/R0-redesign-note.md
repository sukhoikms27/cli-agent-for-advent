# R0. Redesign Note — agent-driven расписание (подписки через MCP-tools)

> Переработка задач Day 18 (05/06) в ответ на критику: расписание должно **задавать сам агент**,
> а не оператор через env-переменные. Развёрнут поверх уже реализованного Day 18 (задачи 01–12 green).

## Что было не так (исходная реализация)

`WeatherScheduler` (задача 05) + wiring (06) конфигурировались **оператором** при старте сервера:
- `CLI_AGENT_WEATHER_CITIES` — фиксированный список городов
- `CLI_AGENT_WEATHER_INTERVAL_SECONDS` — один общий интервал для всех
- Агент мог только разово `collect_weather` — не управлять расписанием.

Это противоречит:
1. **Подсказке куратора** (`00-task.md`): «У mcp есть понятие tool. Покопай в эту сторону» —
   расписание выражается через tool'ы, которыми оперирует **агент**.
2. **Смыслу задания** «24/7 агент»: агент (через сервер) сам решает, что и как часто собирать.
3. **Требованию пользователя**: «Все эти данные (город, интервал) должен отдавать агент, а не мы хардкодить».

## Новая модель — agent-driven подписки

Агент в REPL управляет расписанием через **MCP-tools**:
- `subscribe_weather(city, interval_minutes)` — зарегистрировать periodic-сбор.
- `list_weather_subscriptions()` — активные подписки.
- `unsubscribe_weather(city)` — остановить.

Сервер хранит подписки в реестре и крутит **per-city background-jobs** 24/7 — независимо от запросов.
Результат забирается существующим `get_weather_summary` (pull-модель).

## Модель доставки — pull, не callback

**Почему callback невозможен в нашем CLI-агенте:**
- MCP-tools — синхронный **request/response** (агент спрашивает → сервер отвечает). Сервер не может
  «позвать обратно» tool-loop агента, чтобы сам начать взаимодействие.
- MCP определяет server→client notifications (`notifications/resources/updated` и т.п.), но…
- **CLI-агент реактивен**: он живёт только когда пользователь печатает в REPL. Между вводами нет
  фонового слушателя, который бы отреагировал на входящее уведомление и показал сводку сам. Даже если
  сервер отправит «новые данные готовы», агенту некому это отобразить, пока пользователь не напишет.

**Где живут «24/7» и «периодичность»:**
В **сервере**, не в агенте. Background-scheduler крутится непрерывно и собирает погоду по подпискам
раз в N минут — независимо от того, спрашивает кто-то или нет. Роль агента:
- **Инициировать** подписку (`subscribe_weather`) — задать расписание.
- **Забрать** результат по запросу (`get_weather_summary`) — «следующий запрос».

Это точно ложится в задание: «агент работает 24/7» = серверный компонент крутится 24/7 и периодически
копит; сводка доступна при запросе.

## Ответы на развилки (решения пользователя)

| Развилка | Решение | Обоснование |
|---|---|---|
| Персистентность подписок | **В памяти** | Рестарт сервера → пусто, агент перерегистрирует. Достаточно для демо «периодической работы локально»; проще, без +1 storage-сущности. |
| Env `CLI_AGENT_WEATHER_*` | **Убрать полностью** | Единственный путь конфигурации = tools. Нет двух точек конфигурации (env + tools), чище. |
| Гранулярность интервала | **Per-city** | `subscribe_weather(city, interval_minutes)` — каждый город со своим. Гибче, естественнее для agent-driven. |
| Первый замер | **Ждать интервал** | `delay` ПЕРВЫМ в per-city job: подписался → через N минут первый сбор. Строго «раз в N». |

## Что меняется vs что остаётся

### Меняется
| Было | Стало |
|---|---|
| `WeatherScheduler(client, store, cities, interval, dispatcher)` — фиксированный конф | `WeatherScheduler(client, store, scope, dispatcher)` — пустой реестр + `subscribe`/`unsubscribe`/`list` |
| `start(scope)` — стартует общий loop | `subscribe(city, interval)` — per-city job динамически |
| `CLI_AGENT_WEATHER_CITIES`/`_INTERVAL_SECONDS` env | **убрать** — конфигурация только через tools |
| `McpServerApp.main()` создаёт scheduler с автоподписками | пустой scheduler, проброс scope, `stopAll()` на shutdown |
| delay ПОСЛЕ collect (первый замер немедленно) | delay ПЕРВЫМ (первый замер через интервал) |
