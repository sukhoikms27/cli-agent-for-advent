# Задача 00. Задание курса — День 18. Планировщик и фоновые задачи

**Источник:** `plan/newdays/day18.md`.

## Задание курса

> **День 18. Планировщик и фоновые задачи**
>
> Сделайте MCP-инструмент с отложенным или периодическим выполнением.
>
> Пример:
> - 👉 reminder
> - 👉 периодический сбор данных
> - 👉 регулярный summary
>
> Инструмент должен:
> - 👉 сохранять данные (JSON / SQLite)
> - 👉 выполняться по расписанию
> - 👉 возвращать агрегированный результат
>
> **Результат:** Агент, который работает 24/7 и периодически выдаёт сводку.

## Контекст предыдущих дней

- **Day 16** — MCP-клиент (stdio), `tools/list` без вызова tools.
- **Day 17** — собственный MCP-сервер над GitHub REST (read-only tool `get_repo`),
  полный LLM function-calling цикл в агенте (`ContextAwareAgent.runToolLoop`, `MAX_TOOL_ROUNDS=4`),
  `McpToolExecutor`/`ToolExecutor`, OpenAI tool-модели. Итоги: `plan/finisheddays/day-17/day17-results.md`.
- **Day 17-vps** — remote Streamable HTTP transport: dual-mode сервер (env `CLI_AGENT_MCP_MODE=http|stdio`),
  `mcpStreamableHttp` на `embeddedServer(CIO)`, bearer-auth, Shadow fat-jar, деплой на VPS
  (systemd + nginx TLS + certbot). Итоги: `plan/finisheddays/day-17-vps/vps-deploy.md`.

**Ключевое ограничение текущего сервера:** он **чисто синхронный** — один tool `get_repo`
вызывается LLM в момент диалога. Нет ни планировщика, ни персистентного хранилища, ни фоновых
задач. День 18 вводит всё это.

## Уточнение куратора (развилка «где живёт расписание»)

> `[24.06.2026 16:40] Алексей Гладков`: У mcp есть понятие tool. Покопай в эту сторону — если этого
> не достаточно для полноценного ответа тебе — объясни детально различие между этими двумя вариантами.

**Принятая трактовка:** «выполняться по расписанию» выражается **через MCP-tools** — есть явный
on-demand tool `collect_weather` (модель может сама периодически дёргать сбор), **и** серверный
background-loop (задачи 05–06), который автоматически копит данные раз в N часов. Так «по
расписанию» достигается обоими способами:

1. **Автоматически** — `WeatherScheduler` (background coroutine в долгоживущем http-процессе на VPS).
2. **Через tool-concept** — `collect_weather`/`get_weather_summary`, которые LLM может вызывать
   по своему усмотрению (включая «приходи раз в час и собери»).

Это снимает дилемму «сервер сам vs агент по таймеру»: оба слоя присутствуют, и каждый — это
полноценный ответ. Демонстрация (задача 13) показывает периодическую работу локально на http-сервере
с интервалом 60с, без обязательного деплоя на VPS для зачёта.

> **⚠️ ПЕРЕРАБОТАНО (R0–R5).** Исходная реализация (задачи 05/06) задавала расписание **оператором**
> через env (`CLI_AGENT_WEATHER_CITIES`/`_INTERVAL_SECONDS`). После критики пользователя («город и
> интервал должен отдавать агент, а не мы хардкодить») и в развитии подсказки куратора расписание
> перепроектировано в **agent-driven**: агент регистрирует periodic-сбор через MCP-tools
> (`subscribe_weather`/`list_weather_subscriptions`/`unsubscribe_weather`), сервер хранит per-city
> jobs в реестре (в памяти), интервал per-city, первый замер через интервал. Модель доставки — pull
> (callback невозможен в реактивном CLI-REPL: MCP-tools синхронны). Подробности —
> [`R0-redesign-note.md`](./R0-redesign-note.md). Итог: **7 tools**, env убран.

## Принятые решения пользователя (развилки)

1. **Объём дня (scope):** сбор погоды + summary. Reminder (отложенные задачи, второй пример
   задания) **не** делается — отнесено в Day 18+.
2. **Архитектура:** «чистая архитектура с возможностью расширения» — мономолитный
   `GitHubMcpServer.kt` (303 строки) разбивается по пакетам `util/`, `tools/`, `weather/`; сервер
   становится мульти-tool (точка расширения — `McpServerFactory`). См. задачу 01.
3. **API-таргет:** **Open-Meteo** (погода) — бесплатно, без API-ключа, read-only. Вариант #2 из
   day17-results. Идеально ложится в «периодический сбор данных» + «регулярный summary».
4. **Scheduler:** выражается через tools (on-demand `collect_weather`) + серверный background-loop
   (задачи 05–06) — по подсказке куратора.
