# Week 04 — MCP: подробный конспект

## Короткое резюме

Model Context Protocol (MCP) — открытый стандарт взаимодействия AI-приложений с внешними
инструментами и источниками данных. Он вводит единый слой между host-приложением с LLM и
разнородными сервисами: таск-трекерами, репозиториями, базами знаний, календарями, CI/CD и
корпоративными системами. Сервер MCP описывает доступные tools через schemas, принимает вызовы в
формате JSON-RPC и преобразует их в операции конкретного API, CLI или другого backend.

Ценность MCP — в совместимости, повторном использовании интеграций и контролируемом наборе
операций. Цена этой универсальности — дополнительная инфраструктура, поверхность атаки и расход
контекста на tool schemas и результаты вызовов. Для локальных задач с готовым CLI связка
Skill + CLI часто проще и экономнее, особенно при длинной последовательности действий. Для
удалённых, публичных и корпоративных интеграций MCP остаётся практичным стандартом.

## Главная идея лекции

LLM сама по себе знает только то, что было доступно во время обучения и передано в текущем
контексте. Реальная работа требует доступа к актуальным данным и действиям: прочитать задачи,
найти код, проверить календарь, запросить внутреннюю базу знаний, запустить автоматизацию или
получить состояние сложной корпоративной системы.

Если каждое AI-приложение вручную интегрировать с каждым сервисом, число связей быстро растёт.
MCP предлагает общий контракт: AI host понимает протокол, а MCP server адаптирует этот контракт к
конкретной системе. Интеграцию можно один раз реализовать и затем подключать к разным клиентам и
моделям, если они поддерживают соответствующий MCP transport и возможности протокола.

## Зачем появился MCP

Обычные модели хорошо работают с общими знаниями, но не знают внутреннего состояния организации.
Ценные данные часто распределены между корпоративными wiki, чатами, таск-трекерами,
репозиториями, файловыми хранилищами и специализированными приложениями. Они могут быть плохо
структурированы, но всё равно содержать решения, договорённости и историю работы.

Второй класс задач связан не с чтением, а с действиями. AI может потребоваться запустить pipeline,
создать задачу, получить метрики, сформировать отчёт или управлять календарём. Без стандартного
интеграционного слоя каждый агент должен знать особенности каждого API: authentication, endpoints,
форматы ошибок, pagination, ограничения и схемы данных.

MCP уменьшает эту связанность. Сервис предоставляет понятные модели tools, resources или prompts,
а host получает единый способ обнаружить возможности и вызвать нужную операцию.

## MCP как стандарт, а не framework

MCP определяет правила взаимодействия, сообщения и роли компонентов. Он не привязан к одному
языку программирования или одной библиотеке. SDK для Python, Kotlin, TypeScript и других языков —
это реализации стандарта, а не сам стандарт.

Это различие важно архитектурно:

- совместимость определяется соблюдением протокола, а не выбором конкретного SDK;
- client и server могут быть написаны на разных языках;
- transport и deployment выбираются под задачу;
- библиотеку можно заменить, не меняя смысл публичного MCP-контракта.

Framework обычно задаёт способ организации приложения. MCP задаёт границу взаимодействия между
AI host и внешней возможностью.

## MCP и корпоративные системы

В компании MCP может стать унифицированным входом к нескольким классам систем:

- таск-трекерам и системам управления проектами;
- репозиториям, code review и build-инфраструктуре;
- внутренним файлам, wiki, чатам и базам знаний;
- DevOps, CI/CD, мониторингу и операционным процедурам;
- 1C, SAP, CRM и другим системам со сложным пользовательским интерфейсом;
- календарям, почте и персональным рабочим данным.

Естественный язык становится более простым интерфейсом к сложной предметной области. Однако MCP
не отменяет бизнес-правила и права доступа. Агент должен получать только те tools и данные, которые
нужны его роли, а критические операции должны иметь validation, audit и при необходимости
подтверждение человеком.

## MCP vs API

MCP не заменяет API. API остаётся интерфейсом конкретного сервиса, а MCP обычно располагается над
ним как адаптер для AI-клиентов.

Разница состоит в назначении:

- API описывает, как программный клиент обращается к конкретной системе;
- MCP описывает, как AI host обнаруживает и вызывает предоставленные возможности;
- MCP server переводит tool call в HTTP request, CLI-команду, запрос к базе или внутреннюю функцию;
- результат backend преобразуется в формат, понятный MCP client и модели.

До появления общего слоя каждое AI-приложение строило отдельные интеграции с каждым API. После
введения MCP множество клиентов могут использовать один серверный адаптер. При одном локальном
агенте и одном простом API такой слой может быть лишним; при нескольких клиентах и повторном
использовании он даёт больше пользы.

## Как устроен MCP

Базовая цепочка включает несколько ролей:

1. Пользователь работает с интерфейсом или CLI.
2. Host-приложение управляет агентом, контекстом и вызовами LLM API.
3. MCP client внутри host поддерживает соединение с MCP server.
4. MCP server публикует возможности и обрабатывает вызовы.
5. Конкретный tool обращается к API, локальной программе или другому backend.
6. Результат возвращается агенту и включается в следующий sampling call.

Локальный server может запускаться как subprocess и общаться через stdio transport. Remote MCP
Server размещается в доступной сети и использует поддерживаемый удалённый transport. Поэтому
JSON-RPC — формат сообщений протокола, а HTTP или stdio — способы их доставки; эти понятия нельзя
считать взаимозаменяемыми.

## Tools, schemas и JSON-RPC

Tool — именованная операция, доступная агенту. У неё есть description и input schema. Schema
описывает аргументы, типы и обязательные поля. По этим данным модель решает, подходит ли операция
для запроса, и формирует structured arguments.

JSON-RPC 2.0 задаёт форму request-response сообщений: версию протокола, id, method, params, result
или error. Например, client может запросить список tools, а server вернуть описания операций и их
schemas. Затем client вызывает выбранный tool с аргументами.

Качество schema влияет сразу на три свойства:

- точность выбора операции;
- безопасность и возможность validation до выполнения;
- объём контекста, который участвует в sampling calls.

Description должно быть достаточно конкретным, но не перегруженным. Параметры следует ограничивать
типами и required-полями. Tool handler не должен доверять аргументам только потому, что их
сформировала модель.

## Как подключается MCP-сервер

Для готового AI-клиента подключение обычно сводится к регистрации server и его transport. Remote
вариант требует адрес, authentication и доступность сервера. Локальный вариант может описывать
command и args, которыми запускается subprocess.

Remote MCP Server требует:

- размещённое серверное приложение;
- доступ к реальным API tools или внутренним функциям;
- реализацию MCP-протокола;
- сетевую публикацию и TLS для внешнего подключения;
- authentication, authorization, logging и ограничения операций;
- регистрацию в custom tools или connectors используемого host.

Для Claude Desktop локальные servers задаются в `claude_desktop_config.json` через объект
`mcpServers`. Конфигурация связывает логическое имя сервера с командой запуска и аргументами,
например с путём к собранному JAR.

## Пример реализации MCP-сервера

Пример на Kotlin показывает адаптер прогноза погоды. Проект подключает Model Context Protocol
Kotlin SDK, logging, Ktor client и JSON serialization. Создаётся server instance с именем, версией и
server options, после чего настраивается stdio transport.

Отдельный `HttpClient` работает с `api.weather.gov`. Он задаёт базовый URL, headers, content type и
`ContentNegotiation`, а сериализация игнорирует неизвестные поля. Функция прогноза сначала получает
metadata точки по latitude и longitude, затем обращается к найденному forecast endpoint и
формирует читаемый результат.

Ключевой элемент — регистрация `get_forecast` через `server.addTool`:

- name идентифицирует операцию;
- description объясняет модели назначение;
- input schema задаёт числовые `latitude` и `longitude`;
- required list требует оба аргумента;
- handler извлекает и валидирует значения;
- handler вызывает HTTP client;
- ответ преобразуется в `CallToolResult` с `TextContent`.

Здесь происходит основной mapping: AI вызывает MCP tool, а handler переводит вызов в конкретный
HTTP request. SDK обслуживает протокольную часть, но семантику операции, validation и доступ к
backend проектирует разработчик.

## Плюсы MCP

Первое преимущество — прямое использование внешних возможностей из AI host. Пользователь задаёт
цель, агент выбирает подходящий tool, получает данные и объединяет результаты нескольких систем.

Второе — изолированный доступ. Server может предоставить только чтение календаря, не предоставляя
создание событий. Это не полная модель безопасности, но важная граница capability-based access:
непубликованную операцию нельзя вызвать через данный контракт.

Третье — единый стандарт. Клиентская часть не должна заново изучать уникальные соглашения каждого
API. Готовый MCP server можно повторно использовать с несколькими совместимыми hosts.

Дополнительные плюсы — обнаружение возможностей через schemas, независимость языков реализации и
возможность централизованно обновлять интеграционный слой.

## Минусы и риски MCP

Tool access создаёт реальную поверхность атаки. Входные данные могут содержать prompt injection,
вредоносную инструкцию или значение, которое опасно интерпретировать как shell command. Если
handler без validation передаёт строку в shell или критический API, возникает риск destructive
action.

Практические меры защиты:

- минимальный набор tools и least privilege;
- строгая validation аргументов и allowlists;
- отказ от shell interpolation;
- read-only режим по умолчанию;
- подтверждение критических действий;
- sandbox, timeouts и resource limits;
- audit log и наблюдаемость;
- раздельные credentials для разных scopes;
- недоверие к результатам внешних tools как к инструкциям.

Второй недостаток — стоимость разработки и эксплуатации server. Для личного локального сервиса с
одним клиентом прямой API или CLI может быть проще. MCP оправдан, когда стандартизация и повторное
использование перекрывают стоимость дополнительного слоя.

## MCP vs Skill + CLI

MCP публикует формальные tools через протокол. Skill + CLI передаёт модели инструкции по работе с
существующей командной утилитой, а фактический вызов выполняется через shell tool.

MCP удобнее, когда нужен удалённый стандартный endpoint, несколько клиентов, формальные schemas и
интеграция с hosts без локальной установки специального CLI. Skill + CLI удобнее, когда утилита уже
установлена рядом с агентом, команды легко объединять, а инструкции можно загружать только по
необходимости.

Оба подхода решают похожую задачу — дают модели управляемый интерфейс к внешней системе. Выбор
определяется topology, security model, доступностью CLI, повторным использованием и стоимостью
контекста.

## Token flow и стоимость MCP

Токены расходуются не на локальную передачу между UI и host, а на sampling calls к LLM. В prompt
могут входить system instructions, user input, история, tool schemas и предыдущие tool results.
После вызова tool модель получает результат и делает следующий sampling call, куда часто снова
попадает накопленный контекст.

Иллюстративный расчёт использует system prompt около 1500 токенов, user prompt около 200, schemas
шести tools суммарно около 5400, tool result около 3500 и response около 600. Уже первый input при
таких допущениях составляет примерно 7100 токенов. Следующие шаги добавляют предыдущую историю и
результаты.

Точные итоговые числа в исходном объяснении неоднозначны и частично корректируются по ходу речи,
поэтому их нельзя считать benchmark. Надёжный вывод другой: полезный пользовательский текст может
занимать малую долю общего контекста, если schemas велики, результатов много, а sampling calls
повторяются.

Три основных источника overhead:

- **idle overhead** — schemas доступны в контексте даже когда конкретная возможность не нужна;
- **batching** — длинная последовательность действий накапливает историю calls и results;
- **schema weight** — большое число подробных tools увеличивает каждый релевантный prompt.

Утверждение о «квадратичном росте» следует воспринимать как описание наблюдаемого сценария с
повторной передачей растущей истории, а не как универсальную математическую гарантию протокола.
Фактическая стоимость зависит от host, модели, caching, compacting и стратегии формирования prompt.

## Когда Skill + CLI выгоднее

Skill + CLI особенно полезен, когда:

- агент и утилита работают на одной машине;
- CLI уже стабилен и имеет хорошую документацию;
- инструкции можно загрузить on demand;
- требуется выполнить много однотипных действий;
- shell-команды можно объединить в один вызов;
- нулевой idle overhead важнее унификации между hosts.

В рассмотренном примере MCP server предоставляет 46 tools. Их schemas загружаются как постоянный
набор возможностей, тогда как `SKILL.md`, references и bash schema активируются при выборе навыка.
Разница в размере самих описаний может быть умеренной; существеннее то, как часто они повторно
попадают в sampling calls и можно ли batch-ить операции в CLI.

## Почему MCP всё ещё нужен

Тезис `MCP deprecated` в лекции — провокационное сравнение, а не объявление об отмене протокола.
Skill + CLI не является универсальной заменой.

MCP сохраняет ценность в следующих случаях:

- server находится удалённо;
- интеграцией пользуются разные AI hosts;
- компания публикует единый управляемый контракт;
- локальный CLI отсутствует или его установка нежелательна;
- нужны public tools для внешних пользователей;
- требуется отделить credentials и выполнение от машины пользователя;
- важны формальные schemas и централизованное управление.

Пока стоимость токенов приемлема, многие команды предпочтут стандартную интеграцию собственной
CLI-обвязке. Инженер должен знать оба подхода и измерять их на своём workload.

## Практическая ценность для работы

Внутри компании полезный путь начинается с ограниченного use case: read-only доступ к базе знаний,
поиск задач или получение состояния pipeline. После validation можно расширять набор tools и
подключать новые команды.

Такой проект может стать основой корпоративного AI-инструмента и центра компетенций. Его ценность
нужно показывать через измеримые эффекты: сокращение времени поиска, уменьшение ручных переходов
между системами, долю успешных операций, token cost, количество отказов validation и отсутствие
несанкционированных действий.

Технический успех недостаточен без обучения пользователей. Нужны документация, примеры безопасных
запросов, ownership интеграций и понятный процесс добавления новых возможностей.

## Термины

- **MCP** — Model Context Protocol, стандарт взаимодействия AI hosts с внешними возможностями.
- **Host** — приложение, которое управляет агентом, LLM и MCP clients.
- **MCP client** — компонент host, поддерживающий соединение с конкретным server.
- **MCP server** — процесс или сервис, публикующий tools, resources и prompts.
- **Tool** — вызываемая операция с именем, description и input schema.
- **Tool schema** — формальное описание аргументов операции.
- **JSON-RPC** — формат request-response сообщений, используемый протоколом.
- **stdio transport** — обмен сообщениями с локальным subprocess через standard input/output.
- **Remote MCP Server** — сетевой server, доступный удалённому host.
- **Sampling call** — обращение к LLM для следующего шага генерации или выбора tool.
- **Tool result** — данные, возвращённые после выполнения операции.
- **Idle overhead** — стоимость доступных описаний tools без их полезного применения.
- **Schema weight** — вклад schemas в размер prompt.
- **Batching** — объединение нескольких действий для сокращения числа отдельных шагов.
- **Skill + CLI** — инструкции для модели и выполнение действий через командную утилиту.
- **Kotlin SDK** — реализация MCP для Kotlin, использованная в примере.
- **Claude Desktop config** — конфигурация локальных MCP servers в Claude Desktop.

## Что важно запомнить

1. MCP — стандарт, а не framework и не замена API.
2. Server преобразует унифицированный tool call в конкретное действие backend.
3. JSON-RPC описывает сообщения; stdio и remote HTTP относятся к transport.
4. Tool availability должна следовать принципу least privilege.
5. Schemas и results влияют на token cost каждого sampling call.
6. Skill + CLI часто эффективнее для локальных и пакетных операций.
7. MCP остаётся полезным для удалённых, публичных и корпоративных интеграций.
8. Решение нужно принимать по измерениям, а не по лозунгу о превосходстве одного подхода.

- Проверить rejection отсутствующих и неверных аргументов.
