# Summary вводного занятия AI Advent Challenge #8

## Назначение занятия

Это вводное занятие фиксирует организационные правила AI Advent Challenge #8: формат обучения, порядок выдачи и сдачи заданий, условия выбывания, рекомендации по инструментам, ожидания по работе в чате и базовые требования к участникам.

Главная идея курса: участник должен не просто писать код, а научиться эффективно использовать AI-инструменты, harness/code-assistant среды и агентные подходы для решения практических задач.

---

## Ключевой вывод

Challenge построен как интенсив на 7 недель, где каждый рабочий день участник получает практическое задание. Для прохождения необходимо сдавать все 5 заданий каждой недели. Проверяется прежде всего работающий результат, подтвержденный видео, а не красота или ручное качество кода.

Организаторы явно рекомендуют максимально опираться на AI-инструменты: Codex, Claude Code, Cursor, OpenCode и другие harness/code-assistant решения. Цель — научиться ставить задачи нейросетям, управлять их работой и быстро получать работающие прототипы.

---

## Формат обучения

| Параметр | Правило |
|---|---|
| Общая длительность | 7 недель |
| Структура | Каждая неделя разбита на 5 учебных дней |
| Тематика | Каждая неделя посвящена отдельной теме |
| Выдача заданий | Каждый будний день |
| Время выдачи заданий | 14:00 |
| Выходные | Суббота и воскресенье без новых заданий |
| Сложность | От простого к сложному: внутри недели и по ходу всего курса |
| Самое простое задание недели | Обычно понедельник |
| Самое сложное задание недели | Обычно пятница |

Задания можно выполнять в день выдачи или закрывать пачкой на выходных. Однако организатор прямо рекомендует не откладывать всё на выходные: задания могут оказаться сложнее, чем выглядят на первый взгляд.

---

## Правила заданий

Каждый день участник получает задание с описанием ожидаемого результата. Описание результата является примерным, потому что участникам разрешено выбирать:

- любые языки программирования;
- любые harness/code-assistant инструменты;
- любые нейросетевые модели;
- любые технические стеки;
- разные форматы реализации: сайт, мобильное приложение, Telegram-бот, backend-сервис и т.д.

Цель — получить разнообразие решений и коллективный опыт по разным инструментам и моделям.

---

## Формат сдачи

Основной формат сдачи почти всех заданий:

1. **Видео** с демонстрацией результата.
2. **Код** или ссылка на код.

Ключевой акцент сделан на видео. По словам организатора, именно видео фактически является главным материалом для проверки, потому что оно показывает, что решение действительно работает. Код вторичен.

Рекомендуемый порядок сдачи:

1. Записать видео с демонстрацией работающего результата.
2. Подготовить код или репозиторий.
3. Загрузить материалы на Яндекс.Диск, другой файлообменник, GitHub или аналогичный сервис.
4. Внести ссылку в специальную таблицу, ссылка на которую находится под видео вводного занятия.
5. Если участника нет в таблице — написать в чат, чтобы его добавили.

---

## Условия прохождения и выбывания

Главное правило: каждую неделю нужно закрывать все 5 заданий предыдущей недели.

Проверка происходит по понедельникам, когда в 14:00 выдаются новые задания. Участники, которые не сделали все 5 заданий за предыдущую неделю, выбывают.

Организатор подчеркивает, что формат без выбывания уже пробовали, и при таком подходе до конца доходит слишком мало людей. Поэтому правило выбывания является частью механики challenge.

---

## Дедлайны и допуски

Участникам дается целая неделя на выполнение 5 заданий. Поэтому существенных продлений не предусмотрено.

Допустимый люфт по дедлайну:

- максимум **15 минут**;
- просьбы вида «можно ещё 2 часа» не принимаются;
- после превышения допуска участника исключают.

Правила известны заранее, поэтому организатор просит не рассчитывать на индивидуальные исключения.

---

## Главная методическая рекомендация

Организатор настоятельно рекомендует с самого начала «забить на код» в смысле ручного написания и ручного анализа каждой строки. Основной фокус должен быть на использовании harness/code-assistant инструментов.

Практическая установка:

- не пытаться всё писать вручную;
- максимально делегировать реализацию AI-инструментам;
- учиться формулировать задачи;
- учиться управлять AI-кодингом;
- проверять работоспособность через результат;
- сравнивать разные инструменты между собой;
- делиться выводами в чате.

Рекомендуемые инструменты, упомянутые во вводном занятии:

- Codex;
- Claude Code / Claude;
- Cursor;
- OpenCode;
- другие harness/code-assistant решения.

Организатор считает такой подход самым быстрым способом научиться.

---

## Термины: code-assistant / harness и агент

Во вводном занятии отдельно разведены два термина.

### Code-assistant / harness

Это инструмент, который помогает участнику разрабатывать код. Примеры:

- Codex;
- Claude Code;
- Cursor;
- OpenCode;
- другие похожие среды.

Такие инструменты используются участником как помощники в разработке.

### Агент

Агент — это уже часть приложения или сервиса, который разрабатывает участник. В этом случае приложение обращается к API нейросети и предоставляет пользователю AI-функциональность как часть продукта.

Пример: пользователь взаимодействует с сервисом, а внутри сервиса есть вызов API агента или LLM.

---

## Двойная перспектива курса

Многие темы курса применимы в двух плоскостях:

1. **Для собственного harness/code-assistant окружения**  
   Например, настройка правил, state management, cursor rules, codex rules, служебные файлы и т.д.

2. **Для приложения, которое участник разрабатывает**  
   Например, state management внутри backend/API/agent service.

То есть участники одновременно учатся:

- эффективно использовать AI для разработки;
- строить приложения, в которых AI является частью продукта.

---

## VPS, деплой и инфраструктура

Однозначного требования покупать VPS нет. Необходимость зависит от выбранного типа решения.

VPS может понадобиться, если:

- нужно развернуть сайт с публичным URL;
- делается Telegram-бот;
- нужен backend/API;
- требуется внешний доступ к сервису;
- локальное решение неудобно демонстрировать.

VPS может не понадобиться, если:

- решение локальное;
- делается мобильное приложение;
- используется готовая облачная платформа;
- публичный URL обеспечивается другим способом;
- применяется reverse proxy с локальной машины.

Организатору в целом не важно, где именно развернуто решение. Важно, чтобы был демонстрируемый результат, которым можно поделиться.

---

## Локальные модели

Локальные модели, скорее всего, понадобятся ближе к шестой неделе. До этого момента VPS или отдельная инфраструктура, вероятно, не обязательны, но всё зависит от выбранного формата реализации.

---

## Рекомендации по выбору моделей

Для разных сценариев даны разные рекомендации.

### Для API внутри приложения

Рекомендуется брать самые дешевые модели, если они справляются с задачей. В качестве примера упомянут DeepSeek как дешевый вариант на момент записи.

Логика: если модель используется внутри приложения через API, важно контролировать стоимость токенов.

### Для harness/code-assistant разработки

Рекомендуется, наоборот, начинать с более сильных и дорогих моделей, например Claude. Это повышает качество помощи при разработке.

При этом за 7 недель желательно попробовать разные инструменты и модели:

- начать, например, с Claude;
- затем попробовать Codex;
- затем Cursor;
- затем другие решения.

Цель — сформировать личное практическое понимание, какой инструмент в каких задачах работает лучше.

## Оценка бюджета

Ориентировочная стоимость подписок и токенов на момент записи — около **100 долларов** на весь challenge.

Эта оценка не является жестким правилом: фактическая стоимость зависит от интенсивности использования, выбранных моделей и конкретных инструментов.

## Роль чата

Чат — это не соревновательная арена, а кооперативная среда. Участники «играют вместе», а не друг против друга.

Ожидаемое поведение в чате:

- задавать вопросы;
- отвечать другим участникам;
- делиться опытом;
- обсуждать модели и harness-инструменты;
- помогать с проблемами;
- сравнивать подходы;
- использовать чат как коллективную базу знаний.

Организатор будет на связи в чате в течение всех 35 дней challenge. Чат останется доступен и после окончания challenge.

## Практический чек-лист участника

### Перед стартом

- [ ] Проверить, что есть доступ к таблице сдачи заданий.
- [ ] Если в таблице нет участника — написать в чат.
- [ ] Подготовить место для хранения видео и кода: Яндекс.Диск, GitHub, другой файлообменник.
- [ ] Выбрать основной harness/code-assistant инструмент.
- [ ] Настроить минимум один сильный AI-инструмент для разработки.
- [ ] По возможности установить Caveman и AST Index.
- [ ] Проверить, как быстро записывать короткое видео с демонстрацией результата.

### Каждый день

- [ ] В 14:00 получить новое задание.
- [ ] Оценить ожидаемый результат.
- [ ] Реализовать решение с помощью AI-инструментов.
- [ ] Проверить, что результат работает.
- [ ] Записать демонстрационное видео.
- [ ] Загрузить видео и код.
- [ ] Внести ссылку в таблицу.

### В конце недели

- [ ] Проверить, что закрыты все 5 заданий.
- [ ] Убедиться, что все ссылки доступны.
- [ ] Проверить, что видео открываются.
- [ ] Не оставлять финальную проверку на последние минуты.

## Основные риски

| Риск | Как снизить |
|---|---|
| Отложить задания на выходные | Делать задания в день выдачи или хотя бы не копить все 5 |
| Недооценить сложность пятничного задания | Помнить, что пятница обычно самая сложная |
| Не записать видео | Считать видео главным артефактом сдачи |
| Потратить слишком много времени на ручной код | Делегировать реализацию harness/code-assistant инструментам |
| Превысить дедлайн | Закрывать неделю заранее; не рассчитывать на продления |
| Потратить лишние токены | Использовать более дешевые API-модели и плагины оптимизации |
| Не иметь публичной демонстрации | Заранее продумать деплой, URL, reverse proxy или другой способ показа |
| Изолироваться от чата | Активно использовать кооперативный формат |

## Короткая памятка

1. Challenge длится 7 недель.
2. Каждая неделя — 5 заданий.
3. Новые задания выдаются в 14:00 по будням.
4. Нужно сдавать видео плюс код.
5. Видео важнее кода, потому что показывает работающий результат.
6. Все 5 заданий недели должны быть закрыты к следующему понедельнику.
7. Кто не закрыл неделю полностью — выбывает.
8. Максимальный допуск по времени — 15 минут.
9. Главный фокус — учиться работать через AI-harness/code-assistant инструменты.
10. Чат используется как кооперативная среда, а не как соревнование.
11. VPS нужен только если выбранный формат решения требует публичного URL или backend-инфраструктуры.
12. Для API лучше дешевые модели, для разработки через harness — сильные модели.
13. Желательно попробовать несколько инструментов за время курса.
14. Участники, дошедшие до конца, смогут перейти в Advanced Challenge.

## Итоговая интерпретация

Вводное занятие задает не столько академический формат обучения, сколько формат интенсивной практической игры с жесткими дедлайнами. Успешная стратегия — быстро выдавать работающий результат, активно использовать AI-инструменты, не застревать в ручном кодинге, своевременно фиксировать результат на видео и не нарушать недельный ритм сдачи.

Курс следует воспринимать как тренировку навыка управления AI-разработкой: постановка задач, проверка результата, выбор инструментов, работа с агентами, деплой и коллективный обмен опытом.
