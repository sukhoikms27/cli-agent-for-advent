# Summary: неделя 3. Memory, состояние и stateful-агенты

## Главная тема недели

Третья неделя продолжает тему управления контекстом и переносит фокус на memory и состояние ассистента.

Цель недели — перейти от разрозненных сообщений к полноценному stateful-агенту, который помнит профиль пользователя, учитывает этап задачи, соблюдает ограничения и может продолжать работу после паузы.

## Stateless и stateful агенты

**Stateless-агент** отвечает на каждый запрос почти с нуля. Даже если история сообщений где-то сохранена, у агента может не быть долговременного профиля, правил приложения, state machine и инвариантов.

Пример: пользователь просит написать сервис авторизации, но агент не знает, что проект всегда пишется на Kotlin. В результате он может выдать типовое Python-решение, потому что без профиля и ограничений выбирает наиболее распространённый вариант.

Признаки stateless-подхода:

- нет долговременного накопления контекста;
- нет профиля пользователя;
- нет правил и ограничений;
- агент забывает глобальный контекст задачи;
- перепрыгивает этапы;
- даёт примерно один ответ для всех.

**Stateful-агент** хранит состояние и работает в управляемом процессе. Он знает текущий этап задачи, профиль пользователя, ограничения и допустимые переходы. Например, если задано правило «сначала план, потом код», агент не должен сразу писать код по просьбе пользователя, пока план не утверждён.

## Персонализация

Первый блок stateful-агента — персонализация: один и тот же запрос должен давать разные ответы для разных пользователей и задач.

Профиль можно описывать через три группы данных:

- **стиль**: кратко или подробно, формально или разговорно, с примерами кода или без них;
- **constraints**: стек, запреты, правила проекта, доменные ограничения;
- **контекст**: кто пользователь, зачем он использует агента, какой результат нужен.

Профиль можно собирать через стартовое интервью. Например, Telegram-бот при новом user ID сначала выясняет стиль, ограничения и цель пользователя, а затем сохраняет эти данные в Markdown-файлы, базу данных или другое хранилище.

Персонализация применима не только к коду. Агент для изучения языков должен понимать, нужен ли пользователю TOEFL, разговорная практика или базовые фразы для путешествия.

## Хранение и подмешивание памяти

Данные профиля и состояния могут храниться в разных формах:

- Markdown-файлы;
- база данных;
- API-хранилище;
- векторная память;
- глобальный профиль;
- per project, per repo или per file scope.

Перед каждым взаимодействием агент определяет пользователя, загружает нужные данные и подмешивает их в prompt. Обычно такие данные попадают в system prompt, потому что он задаёт поведение сессии.

При создании собственного агента нужно явно решить, что добавлять в prompt:

- общий системный промпт;
- персонализацию;
- summary прошлой сессии;
- состояние текущей задачи;
- инварианты.

Нельзя бездумно добавлять всё в каждый запрос: это переполняет контекст и создаёт шум.

## Task state machine

Второй ключевой блок — task state machine.

Агент должен работать не как набор отдельных prompt/response, а как управляемый процесс. Для задачи нужно определить:

- что считается task;
- какие у неё стадии;
- какие данные сохраняются;
- какие переходы разрешены;
- какие входы и выходы есть у этапов.

Типовые стадии:

- уточнение задачи;
- planning;
- execution;
- validation;
- done.

Стадии можно расширять: отдельно собирать требования, делить execution на backend/frontend, добавлять code review и end-to-end validation. Убирать базовые стадии стоит осторожно.

Allowed transitions добавляют детерминизм. Например, из `planning` можно перейти в `execution`, из `execution` — в `validation` или обратно в `planning`, из `validation` — в `done` или обратно в `execution`.

Промпты с описанием transitions уже помогают, но для жёстких запретов нужен код. Правила в тексте могут потеряться после summary, compacting или из-за противоречивого контекста.

## Сохранение и возобновление работы

Stateful-агент должен сохранять:

- текущую стадию;
- утверждённый план;
- результаты этапов;
- профиль пользователя;
- constraints;
- важный контекст задачи.

Если работа прервалась, через несколько часов агент может загрузить состояние и продолжить с того места, где остановился. Это даёт больше управляемости, чем универсальные кодовые ассистенты из коробки, где task state обычно нужно создавать самостоятельно.

## Инварианты

Третий блок — инварианты: ограничения, которые не должны меняться от запроса к запросу.

Примеры:

- стек и архитектура;
- Kotlin, KMP, Compose, Spring;
- только MVI;
- обязательная ViewModel;
- запрет RxJava или AsyncTask;
- только бесплатные API;
- бюджет;
- уровень пользователя;
- доменные ограничения.

Инварианты можно вставлять в prompt, но лучше также проверять кодом. Response модели фильтруется по списку инвариантов: если нарушений нет, ответ проходит; если есть, агент получает feedback и должен переделать результат.

Это похоже на линтер для требований, выраженных человеческим языком.

## Антипаттерны

Основные ошибки:

- класть всё сохранённое в каждый prompt;
- не делить память и контекст на слои;
- полагаться только на текстовые правила;
- не валидировать переходы state machine;
- хранить инварианты только в Markdown без проверки;
- позволять пользователю случайной просьбой ломать стек, процесс или архитектуру.

LLM старается помочь пользователю и может согласиться нарушить процесс. Поэтому важные ограничения лучше держать и в prompt, и в кодовой проверке.

## Общая архитектура

Простой stateful-агент может работать так:

1. Получить запрос пользователя.
2. Загрузить профиль.
3. Загрузить state machine.
4. Загрузить инварианты.
5. Собрать prompt через prompt builder.
6. Отправить запрос в LLM.
7. Проверить response.
8. При нарушениях сделать retry или попросить исправить.
9. При успешной проверке вернуть ответ пользователю.

Архитектуру лучше строить через абстракции: LLM engine, prompt builder, profile storage, task state storage, invariant checker и response validator. Тогда конкретный OpenAI, DeepSeek, GigaChat или другой provider можно подключать как реализацию интерфейса.

## Практический вывод

Хороший агент появляется не из одного длинного промпта, а из управляемой системы:

- профиль пользователя;
- memory layers;
- task state machine;
- инварианты;
- prompt builder;
- validation;
- сохранение и восстановление состояния.

К концу недели нужно собрать эти части в stateful-агента. Практика должна показать, какие детали были недооценены и где архитектуру нужно уточнить.
