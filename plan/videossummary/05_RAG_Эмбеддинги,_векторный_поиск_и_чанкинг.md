## Что такое RAG

**RAG (Retrieval Augmented Generation)** — паттерн дополнения ответов LLM с помощью внешних знаний прямо во время инференса. Модель получает ответ не только из базы знаний, которую усвоила при обучении, но и из специфических, внутренних знаний конкретной системы.

Ключевое отличие от MCP:
- **MCP** — общение с внешними источниками (сторонние API, инструменты, данные)
- **RAG** — работа с внутренними знаниями организации (базы знаний, документация, конфлюенс)

RAG происходит **во время инференса** — это не вызов внешнего инструмента, а подгрузка данных в контекст модели.

---

## Проблемы без RAG

1. **Неполный контекст** — модели обучены на общих интернет-данных (Библия, Twitter, Reddit, GitHub), которые не имеют отношения к внутренним корпоративным знаниям.

2. **Смешанные данные** — вся история в конфлюенсе перемешана: несколько версий процессов, данные разных команд, устаревшие и актуальные документы. Модель не может их разграничить как человек.

3. **Галлюцинации** — модель может вернуть "среднее по больнице": например, вместо конкретной политики code review вашей команды — общепринятые нормы.

4. **Ложно-положительные ответы** — самое опасное. Не неправильный ответ, а ответ, похожий на правильный. Именно это происходит, когда люди без достаточной компетенции используют ChatGPT для доказательства своей правоты экспертам.

## Способы улучшения LLM (общая картина)

| Подход | Что делает | Примеры |
|---|---|---|
| **Промпт-оптимизация** | Прямое указание через контекст | MD-файлы, скиллы, субагенты |
| **RAG** | Контекстная оптимизация через внешние знания | Векторные БД, эмбеддинги |
| **Fine-tuning** | Изменение весов модели | Обучение на специализированных данных |

Все три метода **аддитивны**, не взаимоисключающие. Современные LLM (ChatGPT и др.) используют все три одновременно.

---

## Пайплайн RAG

**Без RAG:** запрос → модель → ответ из общей тренировочной базы (возможна галлюцинация)

**С RAG:**
1. Пользователь задаёт вопрос
2. Модель анализирует запрос и определяет пробелы в знаниях
3. Запрос обращается к **векторной базе данных** с документами
4. Находятся семантически похожие чанки текста
5. Чанки **комбинируются с промптом**
6. Модель возвращает ответ с ссылкой на источник

Пример хорошего RAG-ответа:
> "Согласно политике code review ([ссылка на Confluence]), требуется минимум 2 аппрувала до мержа, максимальное время ревью — 24 часа."

## Эмбеддинг-модели vs Генеративные модели

| Тип | Вход | Выход | Назначение |
|---|---|---|---|
| **Генеративные** (ChatGPT, Claude) | Текст | Текст | Генерация ответов, понимание контекста |
| **Эмбеддинг-модели** (Ollama, nomic-embed-text) | Текст | Вектор (числа) | Поиск семантически похожих текстов |

## Как работают эмбеддинги

**Вектор** — массив чисел, представляющий текст. Например, слово превращается в вектор из 768 или 1024 чисел.

### Косинусное сходство

Для сравнения текстов используется **косинусный индекс**:
- `1` — тексты полностью совпадают по смыслу (близкие)
- `0` — тексты о разном (параллельные, не пересекаются)
- `-1` — тексты противоположны по смыслу

### Аналогия с измерениями

Представьте 3D-пространство: X, Y, Z. Можно понять, какой из объектов ближе, даже если они расположены по-разному. Эмбеддинги работают так же, но в **768 или 1024 измерениях** — это позволяет тонко различать миллионы вариаций слов и фраз.

Чем больше размерность — тем меньше "коллизий" (как в хэш-функциях): близкие по смыслу тексты будут близки в пространстве, далёкие — далеки.

### Процесс создания эмбеддинга (pipeline)

1. **Токенизация** — текст разбивается на токены
2. **Embedding layer** — каждому токену присваивается вектор
3. **Self-attention** — модель анализирует взаимосвязи между словами внутри предложения
4. **Pooling** — усреднение векторов для получения единого вектора предложения
5. **Сравнение** — полученный вектор сравнивается с сохранёнными в базе

## Чанкинг (разбивка текста)

Перед векторизацией текст нарезается на **чанки** — фрагменты, которые будут сохранены в базе и искаться по запросу.

### Метод 1: Фиксированный размер
- Режем текст каждые N токенов (например, 500)
- **Минус:** можно разрезать посередине предложения и потерять контекст

### Метод 2: Семантический
- Режем по смысловым блокам (абзацы, разделы)
- Более точный, но сложнее реализовать

### Overlap (перекрытие)
Решение проблемы потери контекста на границах чанков:

```
Без overlap: [1-500] [501-1000] [1001-1500]
С overlap:   [1-500] [451-950]  [901-1400]
```

Чанки частично перекрываются, граница всегда "на замке". Оптимальные параметры (размер чанка, процент overlap) — **экспериментально**, зависят от конкретных данных и задач.

Рекомендация: размер чанка **500–1000 токенов**.

## Инструменты: Ollama для эмбеддингов

**Ollama** — рекомендуемый инструмент для локальных эмбеддингов:

- **Безопасность** — данные не покидают компьютер (приватность)
- **Стоимость** — нулевая (кроме железа)
- **Производительность** — эмбеддинговые модели небольшие (768–3000 параметров, не миллиарды)
- **Совместимость** — можно переключиться на облачный эмбеддер в любой момент
- **Тестирование** — удобно проверять механизмы RAG на малых объёмах

Установка (macOS):
```bash
brew install ollama
ollama pull nomic-embed-text  # или другая модель
```

Работа через HTTP (локально):
```
POST http://localhost:11434/api/embeddings
```

Если деплоить на VPS — просто заменить `localhost` на адрес сервера.

## Реранкинг (Reranking)

**Проблема:** при поиске топ-20 похожих чанков часть результатов может быть нерелевантной (схожая тематика, но не отвечает на вопрос).

Пример: запрос "Как настроить CICD для Kotlin Multiplatform"
- Релевантно: "Настройка GitHub Actions для KMP"
- Нерелевантно: "Обзор CICD инструментов за 2023 год" (тема похожа, ответа нет)

**Реранкинг** — дополнительный этап оценки найденных чанков с помощью **CrossEncoder-моделей**:

1. Ищем топ-20 кандидатов (быстро, миллисекунды)
2. CrossEncoder переоценивает каждый чанк относительно запроса (медленно, секунды)
3. LLM получает отфильтрованные, наиболее точные данные

Реранкинг замедляет пайплайн, но значительно повышает точность.

## Метрики качества RAG

### 1. Качество ретривела
- Доля релевантных найденных чанков (метод "палец вверх/вниз")
- Если много нерелевантных → пересмотр данных или смена модели эмбеддингов

### 2. Правдивость ответов
- Насколько часто модель отвечает из чанков, а не из "общей памяти"
- Индикатор: наличие ссылок на источники в ответе

### 3. Точность финального ответа
- Сравнение с эталонными ответами (можно автоматизировать)
- Набор тестовых вопросов + эталонных ответов → регрессионное тестирование

Минимальная анкета для оценки:
- [ ] Правильные чанки найдены?
- [ ] Ответ основан на контексте (есть ссылки)?
- [ ] Финальный ответ правильный?

## Когда RAG не поможет

| Проблема | Решение |
|---|---|
| Мало данных или данные низкого качества | Fine-tuning |
| Нужна сложная логика ("если данные такие, то сделать так-то") | Промпт-инжиниринг |
| Нужен определённый стиль общения (вежливость, споры, персонализация) | Агенты, инструменты |
| Устаревшие форматы данных (нечитаемые документы) | Мультимодальный RAG, OCR |

RAG хранит **сырые векторизованные данные** для семантического поиска — не более. Он не меняет поведение модели.

---

## Где применять RAG

- **Корпоративные базы знаний** — конфлюенс, вики, чаты, внутренняя документация
- **Прайваси-сценарии** — локальная модель с RAG обогащает ответ глобальной модели в закрытом контуре
- **Специализированные агенты** — медицина, физика, механика, лабораторная диагностика
- **Анализ трендов** — векторизованные данные + аналитический агент
- **Цифровой двойник** — оцифровка переписок, звонков, видео в персональную базу знаний

## Краткое резюме

1. **RAG** — способ дополнить ответы LLM внутренними знаниями организации во время инференса, в отличие от MCP (внешние источники) и fine-tuning (изменение весов).
2. **Эмбеддинги** превращают текст в числовые векторы; косинусное сходство позволяет найти семантически близкие фрагменты — именно на этом строится весь RAG.
3. **Чанкинг с overlap** решает проблему потери контекста на границах фрагментов; оптимальные параметры подбираются экспериментально (обычно 500–1000 токенов).
4. **Ollama** — рекомендуемый инструмент для локальных эмбеддингов: бесплатно, приватно, совместимо с облаком.
5. **Реранкинг** — дополнительный этап фильтрации найденных чанков через CrossEncoder-модели; замедляет пайплайн, но значительно повышает точность.
6. **Метрики RAG** включают качество ретривела, правдивость ответов и точность финального ответа — всё это можно и нужно измерять автоматически.
7. **RAG не универсален**: не поможет при низком качестве данных, сложной логике или нужде в конкретном стиле общения — для этого есть fine-tuning, промпт-инжиниринг и агенты.
