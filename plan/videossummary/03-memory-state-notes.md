# Неделя 3. Memory и состояние ассистента

> Конспект подготовлен по сырой транскрибации видео. Текст отредактирован минимально: исправлены явные ошибки распознавания, орфография, пунктуация и структура. Сохранены исходная логика, формулировки, примеры и акценты лекции.

## 1. Введение: зачем говорить о memory и состоянии

Третья неделя продолжает линию предыдущей недели, где основной темой было управление контекстом. Теперь фокус смещается к memory и состоянию агента.

Дальше курс движется по пути создания собственных агентов. У таких агентов должна быть память и должно быть состояние. Память уже затрагивалась раньше: short-term memory, long-term memory, ветвления и связанные с этим механики. Но здесь память рассматривается в другом контексте: как часть более сложной агентской истории, которую можно внедрять в разные приложения.

Эти идеи применимы не только к кодовым ассистентам. Они нужны для любых помощников:

- отдельных ассистентов внутри приложения;
- кодовых ассистентов;
- сервисных агентов;
- голосовых и чат-ботов;
- специализированных помощников для обучения, доставки, медицины, искусства и других доменов.

Главная задача недели — перейти от набора разрозненных сообщений к полноценному stateful-агенту.

## 2. Stateful и stateless агенты

Агентов можно условно разделить на две группы:

- **stateless agents**;
- **stateful agents**.

Даже если агент уже умеет сохранять контекст в файлах или в другой форме, он всё ещё может оставаться скорее stateless. Он может помнить историю сообщений внутри одной сессии, но при этом у него нет полноценной настройки под пользователя, приложение, ограничения и процесс работы.

### 2.1. Пример stateless-агента

Пользователь просит: «Напиши сервис авторизации».

Если пользователь всегда пишет на Kotlin, но агент этого не знает, он, скорее всего, выдаст распространённое решение на Python. LLM хорошо заточены под Python, поэтому при отсутствии профиля и ограничений модель выбирает наиболее типичный вариант.

Если затем пользователь попросит: «Добавь валидацию токенов», агент просто выполнит текущую инструкцию. Он не спросит, какой результат ожидается, не уточнит процесс, не учтёт долгосрочные правила и не создаст ощущение, что понимает пользователя и задачу.

Такой агент может что-то написать, но результат будет слабым, потому что отсутствуют:

- профиль пользователя;
- правильный технологический стек;
- state machine для работы с запросом;
- обязательные этапы выполнения;
- инварианты, то есть ограничения, которые не должны меняться от запроса к запросу.

### 2.2. Пример с доставкой

Тот же принцип применим к любому агенту.

Если компания работает только в Москве и делает помощника по оформлению доставки, stateless-агент может сказать пользователю, что доставка возможна в Иркутск или Томск. Он не удерживает ограничение бизнеса как постоянное правило.

Для такого сервиса важно, чтобы агент не просто отвечал на текущий вопрос, а учитывал состояние приложения и неизменные ограничения.

### 2.3. Признаки stateless-агента

В рамках лекции stateless-агент определяется так:

- каждый запрос фактически идёт с нуля;
- нет долговременного накопления истории взаимодействия;
- агент забывает глобальный контекст задачи;
- внутри сессии он может помнить локальную историю, но глобальные ограничения после сжатия или новой сессии легко теряются;
- нет правил и ограничений вроде стека, фреймворка, города доставки, запрета на конкретные технологии;
- агент перепрыгивает этапы и делает только то, что пользователь сказал в данный момент;
- нет веса долгосрочной стратегии;
- ответ получается примерно одинаковым для всех пользователей.

Галлюцинации и вариативность ответов всё равно будут, но настоящей персонализации не будет. Разные пользователи с разными инвариантами и профилями не получат действительно разные ответы.

## 3. Что такое stateful-агент

Stateful-агент работает с определённым состоянием.

Например, пользователь планирует сервис авторизации, а в состоянии агента уже есть ограничение: стек — Kotlin и Ktor. Тогда на тот же запрос «Напиши сервис авторизации» агент не бросается сразу писать случайный код. Он учитывает, что работа находится в определённой стадии, предлагает план и помнит выбранный стек.

Если пользователь скажет: «Пропусти план, сразу код», stateful-агент может ответить, что раньше было задано правило: сначала план, затем код, поэтому план нужно утвердить. Это и есть практический смысл state machine.

State machine — основа хорошего агента. Если агент создаётся внутри компании, первым шагом стоит согласовать с менеджерами, разработчиками и другими участниками процесса use case агента и его state machine.

Stateful-агент:

- помнит этап задачи;
- хранит контекст в слоях;
- знает ограничения;
- контролирует переходы между этапами;
- адаптируется под пользователя;
- создаёт ощущение осмысленного продолжения работы.

## 4. Цель недели: от сообщений к stateful-агенту

Глобальная цель недели — сделать из разрозненных сообщений полноценного stateful-агента.

Нужно поэкспериментировать с несколькими строительными блоками:

- персонализацией;
- task state machine;
- инвариантами;
- объединением этих частей в систему.

Эти блоки можно затем использовать как самостоятельные кубики и объединять в более сложные агентские системы.

## 5. Персонализация: один request, разные response

Первый шаг — персонализация.

Идея: один и тот же запрос должен давать разные ответы в зависимости от опыта общения с конкретным пользователем и контекста его задач.

Персонализацию можно описывать через три группы данных:

1. **Стиль**.
   - краткие или подробные ответы;
   - формальный или разговорный тон;
   - с примерами кода или без них;
   - предпочтительный формат объяснения.

2. **Constraints / ограничения**.
   - технологический стек;
   - запрещённые технологии;
   - правила проекта;
   - границы домена.

3. **Контекст**.
   - кто пользователь;
   - зачем он использует агента;
   - какая цель у текущего взаимодействия;
   - как должен выглядеть полезный результат.

### 5.1. Интервью при первом запуске

Один из простых способов собрать персонализацию — сделать этап инициализации агента.

Например, если агент работает через Telegram-бота и видит новый user ID, он может провести базовое интервью:

- какой стиль ответов удобен пользователю;
- какие ограничения нужно соблюдать;
- какой контекст и цель работы;
- какие технологии или подходы важны.

Эту логику можно зашить прямо в промпт: если пишет новый Telegram ID, сначала провести интервью и сохранить данные.

### 5.2. Пример не из кода: изучение языков

Персонализация применима не только к программированию.

Если бот помогает учить иностранные языки, при первом входе он может выяснить:

- какой стиль обучения удобен;
- нужны ли грамматика, разговорная практика или конкретные сценарии;
- какой язык действительно нужен;
- зачем человек учит язык: TOEFL, свободное общение, путешествие, базовые фразы.

Контекст критически важен. Подготовка к TOEFL, разговорная практика и фразы для путешествия требуют разных ответов даже при похожем запросе.

### 5.3. Как хранить профиль

Данные персонализации можно складывать в Markdown-файлы, базу данных, векторное хранилище или другой storage.

Простой вариант:

- отдельные короткие заметки по стилю;
- отдельные constraints;
- отдельный контекст пользователя;
- папка профиля для каждого user ID.

Дальше эти данные подмешиваются в контекст следующей сессии. Так появляется персонализированный агент, который уже не «плавает» в общих ответах.

## 6. Как это делают известные инструменты

Современные инструменты вроде Claude Code, Cursor, ChatGPT и других решают похожую задачу разными способами.

Часто используются Markdown-файлы, но место хранения и scope отличаются:

- база данных;
- Markdown-файлы;
- долговременная память;
- векторное хранилище;
- per project;
- per repo;
- per file;
- глобальный профиль пользователя.

Нельзя сказать, что всегда нужно использовать только per project или только per repo. Способ хранения выбирается под конкретную задачу:

- кодовый ассистент;
- общий сервис;
- голосовой помощник;
- доменный агент внутри приложения.

Почти везде такие данные в итоге попадают в system prompt, потому что system prompt задаёт поведение всей сессии.

## 7. System prompt, summary и конфликты контекста

При создании собственного агента нужно явно решить, что попадёт в system prompt.

В готовых кодовых ассистентах часть этой работы скрыта: инструмент сам сжимает, добавляет правила, подгружает файлы. Но если есть только HTTP-запрос к LLM и собственный агент, эту архитектуру нужно продумать самостоятельно.

В system prompt можно добавлять блоками:

1. общий системный промпт;
2. данные персонализации;
3. summary предыдущего шага или предыдущей сессии;
4. правила текущей задачи;
5. ограничения и инварианты.

Так можно одновременно сохранить персонализацию, учитывать summary и объяснить агенту, что делать с этими данными.

При этом возможны конфликты. Например, после суммаризации диалога нужно собрать новый system prompt, куда попадёт summary предыдущей сессии. Нужно решить, добавлять ли профиль каждый раз или достаточно summary, если в нём уже есть стек и прошлые вводные.

## 8. Общий pipeline персонализации

Архитектура в общем виде одна, хотя конкретные реализации могут отличаться.

Pipeline:

1. Есть хранилище:
   - API на сервере;
   - база данных;
   - Markdown-файлы;
   - векторная память;
   - локальные файлы.

2. Перед каждым взаимодействием агент загружает нужные данные:
   - определяет пользователя по идентификатору;
   - достаёт профиль;
   - достаёт ограничения;
   - достаёт релевантный контекст.

3. Эти данные подсовываются в промпт:
   - в system prompt;
   - в конкретный пользовательский запрос;
   - в отдельный блок после summary.

4. LLM генерирует ответ.

5. Ответ получается персонализированным.

Главная идея: если делается рабочий сервис, а не LLM общего назначения, его ценность будет в персонализации и точном взаимодействии. Иначе сервис мало чем отличается от простой перепродажи чужих токенов.

Будущее LLM-сервисов, агентов и ассистентов — в углублении специализации и персонализации. Пример такого направления — инструменты, которые специализируются на коде, видео или другой конкретной функции, а не обещают «уметь всё».

## 9. Task state machine

Персонализации недостаточно. Агенту желательно не просто задавать вопрос, а организовать работу в task state machine.

Базовую state machine можно расширять, но не стоит без необходимости убирать основные этапы. В лекции подчёркивается, что уменьшать число базовых стадий рискованно, а расширять можно.

Базовые стадии можно понимать так:

- сбор и уточнение задачи;
- planning;
- execution;
- validation;
- done / завершение.

Planning можно расширять: отдельно собирать требования, отдельно обсуждать подход, отдельно проводить консилиум решений. Execution тоже может делиться на разработку backend, frontend, тестирование, ревью и повторную доработку. Validation может быть автотестами, code review или end-to-end проверкой через браузер и Playwright.

Важно не название стадий, а то, что задача движется через контролируемый процесс.

### 9.1. Понятие задачи

Если создаётся собственный сервис, нужно определить понятие task.

Запросы пользователя должны укладываться не просто в отдельные prompt/response, а в понятный task execution:

- что является задачей;
- какие данные у неё есть;
- в какой стадии она находится;
- какие переходы разрешены;
- какие входные и выходные параметры у этапов;
- как подмешивается персонализация.

Тогда каждая задача становится не просто запросом, а персонализированным управляемым процессом.

### 9.2. Allowed transitions

Для state machine нужно задать список разрешённых переходов.

Пример:

- из `planning` можно перейти в `execution`;
- из `execution` можно перейти в `validation` или обратно в `planning`;
- из `validation` можно вернуться в `execution` или перейти в `done`.

Это добавляет детерминизм недетерминированному агенту.

LLM — весовая система и чёрный ящик. Мы не знаем точно, как настроились веса внутри модели, но знаем, как хотим работать с этой моделью. Например, нельзя сначала заселить людей, а потом построить дом. Сначала строится дом, потом люди заселяются. Такие ограничения процесса должны быть заданы явно.

### 9.3. Почему одних промптов мало

Если просто прописать transitions в промпте, это уже даст большой эффект. Это можно попробовать даже в кодовом ассистенте: описать, как задача должна двигаться, какие переходы разрешены, какие стадии обязательны.

Но правила в промпте могут потеряться:

- после суммаризации;
- после компактизации контекста;
- при выходе за пределы окна;
- из-за весовой природы модели;
- из-за противоречивых последующих инструкций.

LLM может нарушать правила так же, как человек может нарушить инструкцию. Поэтому для жёсткого запрета лучше задавать transitions детерминированно в коде.

### 9.4. State в prompt builder

Один из простых способов внедрить state — собрать prompt через prompt builder.

В prompt можно добавить:

- task context;
- текущий state;
- current plan;
- user profile;
- user query;
- правила системы;
- запрет перепрыгивать этапы;
- список допустимых действий текущего step.

В большинстве случаев это даст правильный результат. Агент будет отождествлять себя с текущей стадией и выполнять задачу этой стадии, а не всю задачу целиком.

### 9.5. Более жёсткая изоляция стадий

Если нужно усилить контроль, стадии можно разделить не только промптами, но и на уровне исполнения:

- разные функции;
- разные LLM-сессии;
- разные процессы;
- разные Docker-контейнеры;
- отдельные изолированные агенты.

Каждой стадии можно отдавать не весь план, а только конкретную задачу этого этапа. Программный код в этом смысле более детерминированен, чем текстовая инструкция.

## 10. Сохранение и возобновление задачи

Stateful-агент должен уметь продолжать работу после паузы.

Пример:

1. Пользователь говорит: «Начни сервис авторизации».
2. Агент создаёт утверждённый план.
3. Подмешиваются персонализация и ограничения.
4. Агент начинает выполнение.
5. Работа прерывается: закрыли компьютер, пропал интернет, истекло время, закончились токены.
6. Сохраняются стадия, результат, текущий контекст, запросы пользователя, профиль и другие данные.
7. Через 12 часов всё загружается обратно.
8. Данные снова подаются модели.
9. Агент продолжает: «Мы делали вот это, продолжаем».

Даже простая реализация task state и сохранения состояния может сделать собственного агента сильнее, чем универсальный кодовый ассистент из коробки, потому что универсальные инструменты обычно не хранят такой процесс автоматически. Если нужно, пользователь должен сам создать Markdown-файлы, task state и правила.

## 11. Инварианты

Следующий важный блок — инварианты.

Инварианты — это ограничения, которые не должны меняться от запроса к запросу. В лекции они описываются как то, что запрещено или постоянно ограничено.

Примеры инвариантов:

- стек;
- архитектура;
- язык программирования;
- уровень пользователя;
- бюджет;
- только бесплатные API;
- допустимое количество библиотек;
- запрет на конкретные технологии;
- запрет на RxJava;
- запрет на старые Android-подходы вроде AsyncTask;
- требование использовать Kotlin, KMP, Compose, View для frontend, Spring для backend;
- требование MVI и ViewModel для управления состоянием.

Инварианты полезны даже тогда, когда неизвестно, как именно должно быть сделано приложение. Можно не знать финальную архитектуру, но точно знать, чего делать нельзя.

### 11.1. Инварианты в коде

В коде инварианты можно оформить как абстрактный класс или интерфейс с функцией `check`.

Идея:

1. Есть список инвариантов.
2. Модель генерирует response.
3. Response фильтруется по инвариантам.
4. Если нарушений нет, стадия считается успешно завершённой.
5. Если нарушения есть, агент получает feedback: ответ не проходит по таким-то инвариантам, нужно переделать.

Это похоже на линтер, но для требований, выраженных человеческим языком, а не только кодом.

### 11.2. Инварианты в prompt

Инварианты можно также вставлять в prompt:

- task context;
- invariants;
- user query;
- правило не пересекаться с запрещёнными решениями.

Даже если инварианты просто записаны в Markdown-файлах и подгружаются в кодовый ассистент, это уже сильно повышает продуктивность. Сырая тулза становится заточенной под конкретный рабочий flow.

Но текстовые правила остаются инструкциями. Их можно нарушить. Поэтому лучше иметь двойную защиту:

- инварианты в prompt;
- проверка response кодом;
- хотя бы warning, если агент игнорирует инвариант.

### 11.3. Пример пользовательского запроса, который не должен пройти

Если сервис генерирует мобильные приложения и ориентирован на пользователей, которые плохо разбираются в технологиях, кто-то может попросить сделать решение на Spring Boot и Java просто потому, что где-то это услышал.

Если инварианты говорят, что сервис должен работать в другом стеке, такой запрос не должен пройти. Агент должен предложить разрешённый вариант, а не следовать случайному пожеланию пользователя.

## 12. Антипаттерны

### 12.1. Всё в одном промпте

Плохой подход — всегда пихать в каждый prompt всё, что было сохранено.

Это приводит к переполнению контекста и шуму. Нужно помнить прошлую неделю про управление контекстом и делить данные на слои:

- где-то не нужны инварианты;
- где-то не нужен профиль;
- где-то не нужна история;
- где-то достаточно summary;
- где-то нужны только правила текущей стадии.

Если профиль содержит много указаний для разных типов задач, не нужно добавлять все профили в каждый запрос. Иначе агенту придётся одновременно учитывать программирование, изучение английского и другие несвязанные режимы.

### 12.2. Не валидировать переходы

LLM старается помочь и услужить пользователю. Если пользователь просит пропустить этапы, модель может согласиться.

Если переходы не валидируются, агент может нарушить процесс. Для железного запрета нужен код, а не только текстовая инструкция.

### 12.3. Держать инварианты только в тексте

Инварианты в тексте — это инструкция, а инструкцию можно нарушить. Поэтому лучше проверять их детерминированно, а не надеяться только на prompt.

## 13. Архитектура stateful-агента

В общем виде архитектура может быть простой.

Представим консольный агент в постоянном режиме ввода:

1. Получаем запрос пользователя.
2. Загружаем профиль.
3. Загружаем state machine.
4. Загружаем инварианты.
5. Выбираем, какие данные важны для этой задачи.
6. Собираем prompt через prompt builder.
7. Отправляем prompt в LLM.
8. Валидируем ответ.
9. Если есть нарушения, retry или запрос на исправление.
10. Если validation passed, отдаём ответ пользователю.

Эта архитектура может быть независимой от конкретной модели. Нужно проектировать абстракции:

- интерфейс LLM engine;
- prompt builder;
- profile storage;
- task state storage;
- invariant checker;
- response validator.

Тогда внутрь можно инжектить конкретный OpenAI, DeepSeek, GigaChat или другой provider.

Важна поддерживаемость: если архитектура построена через интерфейсы и не нарушает SOLID, её легко редактировать в будущем. Такой агент может состоять буквально из нескольких файлов, но давать большой эффект.

## 14. Практический пример для рабочего проекта

Если разработчик приходит на Android-проект и хочет внедрить LLM-ассистента, полезно один раз описать правила проекта:

- всегда использовать ViewModel для контроля состояния;
- архитектура только MVI;
- Kotlin;
- Jetpack Compose;
- запрет на неподходящие технологии;
- список вопросов, которые ассистент должен задать до реализации;
- стиль общения как с коллегой;
- локальные паттерны проекта.

Эти правила можно загрузить в state machine и prompt builder. На это нужно потратить время один раз, но дальше ассистент будет работать в нужном flow.

Это не обязательно должен быть большой модуль. Вопрос не в размере кода, а в качестве архитектурной сборки.

## 15. Итоговая цель недели

К концу недели нужно прийти к stateful-агенту.

Отдельные задачи недели будут посвящены:

- персонализации;
- state machine;
- инвариантам;
- объединению персонализации, состояния и ограничений в одну систему.

В теории всё выглядит довольно просто, но практика покажет места, где что-то не предусмотрено, где сложность недооценена и где нужно уточнить архитектуру. Практика остаётся критерием истины, поэтому неделя построена как challenge.
